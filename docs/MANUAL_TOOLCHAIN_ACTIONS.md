## Action 1 — GitHub Personal Access Token (PAT)

### What you must do

Create a GitHub PAT so that:
- The **GitHub MCP server** (`.vscode/mcp.json`) can query issues, PRs, and releases.
- `lib/github.py` can check GHCR manifest availability for update detection.
- CI Docker build/push workflows can push images to GHCR.

### Where to obtain

1. Go to <https://github.com/settings/tokens> (classic) or <https://github.com/settings/tokens?type=beta> (fine-grained).
2. Create a token with the following scopes:
   - `repo` — read repo contents, issues, PRs
   - `packages:read` — read GHCR manifests
   - `packages:write` — push Docker images to GHCR (only if you push images)
   - `workflow` — trigger/read GitHub Actions workflows

### Where to place it

**Option A — .env file (local dev)**
```
# .env (never commit this file)
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

**Option B — OS environment variable (Windows)**
```powershell
[System.Environment]::SetEnvironmentVariable("GITHUB_TOKEN", "ghp_xxx", "User")
```

**Option C — GitHub Actions Secret (CI)**
1. Go to your fork: `Settings → Secrets and variables → Actions → New repository secret`
2. Name: `GITHUB_TOKEN` (auto-provided by GitHub for Actions; no manual step needed for CI)

### What the agent will do once credentials are added

The GitHub MCP server will activate automatically when VS Code loads the workspace.
Agents will be able to: list open issues and PRs, read release notes, check GHCR image availability, and create branches/PRs on your behalf.

## Action 2 — Balena Cloud Account and CLI Login

### What you must do

Create a Balena Cloud account and log in via the CLI to enable fleet deployment.

### Where to obtain

1. Sign up at <https://dashboard.balena-cloud.com/signup> (free for personal use; school fleets may need an organization plan).
2. Install the Balena CLI:
   ```bash
   npm install -g balena-cli
   # OR on Windows:
   winget install Balena.BalenaCLI
   ```
3. Log in:
   ```bash
   balena login
   # Follow the browser OAuth flow
   ```
4. (Optional) For non-interactive CI, generate an API key at:
   `https://dashboard.balena-cloud.com/preferences/access-tokens`

### Where to place it

**Interactive (developer workstation):** `balena login` writes to `~/.balenarc.yml` — nothing else needed.

**CI / non-interactive:**
```
# .env
BALENA_TOKEN=<your-balena-api-key>
```
Or add as a GitHub Actions secret named `BALENA_TOKEN`.

### What the agent will do once credentials are added

The `balena-deploy` skill will be able to: push a new release to the Balena fleet, read device status, and trigger OTA updates.

## Action 3 — Google Analytics Telemetry (Review / Override for School Fork)

### What you must do

The upstream `lib/telemetry.py` contains hard-coded credentials:
```python
ANALYTICS_MEASURE_ID = 'G-S3VX8HTPK7'
ANALYTICS_API_SECRET = 'G8NcBpRIS9qBsOj3ODK8gw'
```
These send anonymized usage data to **Screenly's** Google Analytics property.

For a school fork (Anthias-Bellforge), you must decide:

**Option A — Disable telemetry entirely (recommended for schools)**

Set in your `.env` and in `docker-compose.dev.yml`:
```
TELEMETRY_ENABLED=false
```
Then update `lib/telemetry.py` to check this env var before posting.

**Option B — Use your own GA4 property**

1. Create a GA4 property at <https://analytics.google.com>.
2. Create a Measurement Protocol API secret in GA4 Admin → Data Streams → Measurement Protocol API secrets.
3. Replace the constants in `lib/telemetry.py`:
   ```python
   ANALYTICS_MEASURE_ID = 'G-XXXXXXXXXX'      # your Measurement ID
   ANALYTICS_API_SECRET = 'your-api-secret'
   ```

### Where to place it

Add to `.env`:
```
ANALYTICS_MEASURE_ID=G-XXXXXXXXXX
ANALYTICS_API_SECRET=your-secret
```
And update `lib/telemetry.py` to read from `os.getenv()` instead of hard-coded strings.

### What the agent will do once credentials are added

Telemetry will report to your own GA4 property (or be silenced). The agent can patch `lib/telemetry.py` to read from env vars once you confirm Option A or B.

## Action 4 — Docker Hub Login (for base image mirroring)

### What you must do

The Redis service in `docker-compose.dev.yml` uses `mirror.gcr.io/library/redis:alpine`.
The Jinja2 Docker templates may pull from `docker.io` for base images.
Docker Hub rate-limits unauthenticated pulls (100/6 hours for anonymous IPs).

### Where to obtain

1. Sign up at <https://hub.docker.com/signup> (free tier is sufficient).
2. Log in:
   ```bash
   docker login
   ```

### Where to place it

`~/.docker/config.json` — written automatically by `docker login`.

For CI, add as GitHub Actions secrets:
- `DOCKER_HUB_USERNAME`
- `DOCKER_HUB_TOKEN` (use an access token, not your password)

### What the agent will do once credentials are added

Docker builds will no longer fail with 429 rate-limit errors when pulling base images.

## Action 6 — Redis Connection (local dev, manual start)

### What you must do

The MCP Redis server and the dev stack both require a running Redis instance on `localhost:6379`.

### Steps

Start the dev stack (includes Redis):
```bash
docker compose -f docker-compose.dev.yml up -d redis
```

Or run Redis directly:
```bash
docker run -d --name anthias-redis -p 6379:6379 redis:alpine
```

### What the agent will do once Redis is running

The Redis MCP server will connect automatically and agents will be able to inspect Celery task queues, WebSocket channel-layer keys, and the viewer pub/sub channel.
